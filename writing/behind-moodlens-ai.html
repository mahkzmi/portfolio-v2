Behind MoodLens AI

The engineering story behind building an NLP-powered sentiment analysis application.

Case Study GitHub

I didn't set out to build a machine learning application. I set out to answer a simple question: can a machine understand how we feel?

It started with a personal frustration. I was journaling daily, but I never looked back at what I'd written. Thousands of words, countless emotions — all buried in plain text. I wanted a tool that could surface patterns in my own writing, that could tell me: "You've been feeling anxious lately" or "There's a trend of joy in your entries."

That curiosity turned into a weekend project. And that weekend project became MoodLens AI — a clean, fast, and surprisingly accurate sentiment classifier.

The idea started much smaller

The first version was a Jupyter notebook with a few lines of scikit-learn code. I threw some text at it, got a prediction back, and thought: "This is cool, but no one will ever use this."

So I built a Streamlit interface around it. Then I added confidence scores. Then I cleaned the UI. Then I deployed it. Each step made the project feel more real, more useful. What started as a script became a product.

The evolution

Jupyter Notebook → Streamlit prototype → Polished web app → Open-source project

Finding the right dataset

Data is the foundation of any ML project, and I quickly learned that most public emotion datasets are a mess. Imbalanced classes, inconsistent labels, and text that's cleaned to the point of being unnatural.

I eventually settled on a dataset of social media posts and diary entries labeled with six emotions: joy, sadness, anger, fear, surprise, and disgust. It wasn't perfect, but it was representative of how people actually write — typos, slang, emojis, and all.

Cleaning text is harder than training models

Everyone talks about model selection. No one talks about how much time you'll spend removing emojis, standardizing contractions, and deciding whether "ughhh" should be considered "anger" or just a typo.

Text cleaning is unglamorous, but it's the difference between a model that works and one that doesn't. Here's the core cleaning pipeline I ended up with:

import re
from nltk.corpus import stopwords
from nltk.stem import PorterStemmer

def clean_text(text):
   # Lowercase
   text = text.lower()
   # Remove special characters & numbers
   text = re.sub(r'[^a-zA-Z\s]', '', text)
   # Remove stopwords
   stop_words = set(stopwords.words('english'))
   text = ' '.join([w for w in text.split() if w not in stop_words])
   # Stemming
   stemmer = PorterStemmer()
   text = ' '.join([stemmer.stem(w) for w in text.split()])
   return text

I tried everything — lemmatization, custom stopword lists, spell-checking. In the end, the simplest approach worked best. Over-engineering preprocessing actually hurt accuracy.

Choosing the model: Logistic Regression + TF-IDF

BERT is great. BERT is also huge, slow, and overkill for a real-time sentiment app. I needed something that could run on a modest server and return predictions in under 200ms.

Logistic Regression with TF-IDF vectorization is a classic combination for text classification. It's fast, interpretable, and — with enough data — surprisingly accurate. The feature space is sparse, the model is linear, and the inference is essentially a matrix multiplication.

"The best model is the one that solves the problem, not the one that looks impressive on a leaderboard." — Mahdieh Kazemi

I also experimented with SVM and Random Forest, but Logistic Regression consistently outperformed them in terms of speed without a significant drop in accuracy.

Designing the prediction pipeline

The pipeline needed to be modular, testable, and easy to serialize. I built it using scikit-learn's Pipeline class, which chains the vectorizer and the classifier into a single object.

The pipeline is serialized using Pickle, making it portable and ready for deployment. Loading the model and running a prediction takes less than 200ms on a standard cloud instance.

Making the interface feel human

Machine learning is only half the product. The other half is user experience. I wanted MoodLens to feel like a conversation, not a black box.

That meant:

  • Minimal UI — no clutter, no distractions. Just a text box and a result.
  • Instant feedback — predictions appear as soon as the user stops typing.
  • Confidence scores — users need to know how certain the model is.
  • Clean typography — the result should feel human, not robotic.
Design principle

Every element should serve the user's understanding. If it doesn't help them, remove it.

Mistakes I made

Over-engineering preprocessing

I tried lemmatization, custom stopword lists, and spell-checking. All of them made the model worse. Simplicity won.

Ignoring class imbalance

Some emotions had far more samples than others. I had to oversample the minority classes to get balanced performance.

Deploying without testing

I pushed to production and immediately hit a dependency conflict. Now I always test in a fresh virtual environment.

Not documenting the model

I came back to the project two weeks later and couldn't remember the exact preprocessing steps. Never again.

Lessons learned

1
Technical: Start simple

Begin with the simplest possible solution. Only add complexity when you need it.

2
Product: Design for the user

The best model in the world is useless if no one can understand or trust its output.

3
Process: Document everything

Future you will thank you. Data cleaning, model parameters, deployment steps — write it all down.

What's next

MoodLens AI is just the beginning. I'm planning several major improvements:

  • Transformer models — Fine-tuning a lightweight BERT variant for better accuracy.
  • Emotion history — Track user's emotional patterns over time.
  • Multi-language support — Arabic, French, and Turkish are high on my list.
  • API access — Allow other developers to integrate MoodLens into their own apps.
  • Charts & visualization — Help users see their emotional trends at a glance.

Final thoughts

Building MoodLens AI reminded me why I love engineering. It's not about the latest framework or the most complex model. It's about taking an idea — no matter how small — and turning it into something that works.

It's about asking questions. It's about being curious. And it's about sharing what you learn with others.
