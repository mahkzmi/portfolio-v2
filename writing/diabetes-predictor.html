Behind the Pima Diabetes Predictor

From dirty data to a reliable classifier — the engineering story behind building a diabetes prediction model with XGBoost.

Case Study GitHub

I didn't set out to build a diabetes prediction model. I set out to understand why some machine learning projects succeed and others fail — and the answer, I found, lies in the data.

The Pima Indians Diabetes Dataset is a classic. It's been used in hundreds of tutorials and research papers. But it's also a dataset full of traps: missing values disguised as zeros, imbalanced classes, and features that need careful handling. I wanted to see if I could build a model that actually works — one that could be deployed, trusted, and explained.

This is the story of that journey. The mistakes I made, the lessons I learned, and the model I built.

The data problem: zeros that mean nothing

I loaded the dataset, ran info(), and everything looked clean. No missing values. Perfect, right? Wrong.

In this dataset, a value of 0 in columns like Glucose, BloodPressure, or BMI doesn't mean zero — it means "data not recorded." These were missing values in disguise. And if I treated them as real values, my model would learn nonsense.

# Looking for zeros that should be NaN cols_with_zero = ['Glucose', 'BloodPressure', 'SkinThickness', 'Insulin', 'BMI']
for col in cols_with_zero:
   zero_count = (data[col] == 0).sum()
   print(f"{col}: {zero_count} zeros")

# Output:
# Glucose: 5 zeros
# BloodPressure: 35 zeros
# SkinThickness: 227 zeros
# Insulin: 374 zeros
# BMI: 11 zeros

Over 600 missing values hidden as zeros. If I hadn't caught this, my model would have been a disaster.

Lesson learned

Always, always understand the semantics of your data. Zero is not always zero. Missing values are not always null.

The class imbalance problem

I looked at the target variable and found:

  • 500 healthy patients (65%)
  • 268 patients with diabetes (35%)

This isn't extreme, but it's enough to cause problems. A naive model that always predicts "healthy" would be 65% accurate without learning anything. I needed a metric that reflected the imbalance.

That's why I used F1-Score instead of accuracy. F1-Score balances precision and recall, making it much more robust for imbalanced datasets.

Pipeline architecture: how I built it

I used scikit-learn's Pipeline class to chain all preprocessing and modeling steps into a single, reusable object:

from sklearn.pipeline import Pipeline
from sklearn.impute import SimpleImputer
from sklearn.preprocessing import StandardScaler
from xgboost import XGBClassifier

model = Pipeline([
   ('imputer', SimpleImputer(strategy='median')),
   ('scaler', StandardScaler()),
   ('classifier', XGBClassifier(
     n_estimators=100,
     learning_rate=0.1,
     max_depth=3,
     random_state=42,
     eval_metric='logloss'
   ))
])

This pipeline handles:

  • Imputation — replaces missing values with the median
  • Scaling — standardizes features to mean 0, std 1
  • Classification — XGBoost with early stopping

I optimized the hyperparameters using GridSearchCV with 5-fold cross-validation, scoring on F1. The best parameters were:

  • n_estimators: 100
  • learning_rate: 0.1
  • max_depth: 3

Mistakes I made

Ignoring the zeros

I initially treated zeros as real values. My model performed terribly. It took me an hour to realize the zeros were actually missing values.

Using accuracy as the metric

I kept optimizing for accuracy, but the model was ignoring the minority class. Switching to F1-Score changed everything.

Overfitting the training data

My first model had a depth of 10 and 500 estimators. It was a perfect learner — on the training data. Validation performance was poor.

Not documenting the pipeline

I had to refactor everything when I came back to it. Now I document every decision, even the small ones.

Lessons learned

1
Technical: Data quality is everything

A sophisticated model with bad data will always lose to a simple model with good data.

2
Product: Metrics matter

Choose your evaluation metric based on the business problem, not convenience. F1-Score over accuracy for imbalanced data.

3
Process: Document your reasoning

Why did you choose median imputation? Why XGBoost? Write it down. Future you will thank you.

What's next

This model is just the beginning. I'm planning several improvements:

  • Feature engineering — adding interaction features and domain-specific transformations
  • Model interpretability — using SHAP to explain predictions to end users
  • API deployment — wrapping the model in a FastAPI service for real-time predictions
  • Dashboard — building a simple interface where doctors can input patient data and get results
  • Data augmentation — using SMOTE to handle the class imbalance more effectively

Final thoughts

This project reminded me of something I often forget: machine learning is 80% data, 15% experimentation, and 5% algorithm.

The algorithm matters, yes. But the real work is in understanding your data, cleaning it, and making sure your model is solving the right problem.
